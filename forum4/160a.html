content Bt sync 和 git 结合使用的办法：<br /><br />1. 开发组的每个成员，先各自用git init --bare 初始化一个仓库A.git，<br />2. 再把仓库A.git克隆到Local_A文件夹去。<br />参考命令： git clone /path/to/A.git /path/to/Local_A <br />3. 把A.git文件夹设置为BT Sync的同步文件夹，把只读密钥告诉开发组的其他人。这一步完成后，自己也会收到其他人的密钥。<br />4. 然后，导入其他每个人的只读密钥。<br />5. 在Local_A仓库里面进行设置，把第4步导入所得的文件夹，全部设置为Local_A的远端仓库。<br />6. 把A.git设置为Local_A的默认远端仓库。<br /><br />第6步完成后，你总共应该有N个远端仓库（其中N为开发组成员个数）。设置完成。<br />今后每次使用时，<br />x. 把Local_A仓库的文件夹，定期提交到A.git，即可把更改告知你的团队。<br />y. 从其他组员Sync过来的文件夹（即远端仓库）fetch一下，就可以得知：其他人做了啥更改。<br />z. 你随时可以通过远端仓库，把其他人的更改拉取过来，合并到自己的分支上去，然后自己修改文件，修改完后执行步骤x……。<br /><br />上述这个办法，同样适用于家和办公室之间的重要文件同步。避免了同一文件在两边同时拥有直接写入权限。<br /><br /><br />[b]不适用的情况[/b]：<br />上述办法不适用于百人以上团队，也不适用于GB级别的仓库。这类情况下，应该放弃BT Sync，使用拥有独立ip的服务器作为中心仓库，走常规Git服务器流程。<br /><br />如果实在没有独立ip，那就只能使用共享写入权限的Git/BTSync文件夹，但这种做法十分冒险，最好不要采用，任意一个失误都会导致中心Git仓库丢失更改；任何一个人电脑中毒，甚至会导致全库被毁。<br />更好的解决办法是让Git走BitTorrent等其他P2P协议，然而这类方案似乎还在研发测试阶段，不能使用。
